**Current gamepad mapping (Xbox One S via Bluepad32):**
- Left stick → pan/tilt velocity
- Right bumper (RB) → fine-speed mode (15% scale)
- Left bumper (LB) → latch laser on/off
- Left trigger → laser strobe ~25 Hz while held
- D-pad L/R → pan speed multiplier ±0.25 (range 0.25–10.0, default 4.25×)
- D-pad U/D → tilt speed multiplier ±0.25 (range 0.25–10.0, default 10.0×)
- A → stop all motion
- B → emergency stop (latching)
- Y → home both axes
- Menu → toggle soft limits

**Laser:** GPIO 13 → FET driver gate. Latch = steady on. Trigger = 25 Hz strobe.

**Motion defaults (config.h):**
- Max speed: 180 deg/s, Accel: 30 deg/s², Fine scale: 0.15×
- Soft limits: DISABLED (re-enable after homing configured)
- Pan: ±180°, Tilt: −45° to +90°

**NVS namespace:** `ptz_cfg` — survives flashes. After changing config.h defaults: flash → `reset` → `save` in CLI.

**Key Pi ↔ ESP32 serial protocol facts:**
- Port: `/dev/ttyUSB0`, 115200 baud
- ESP32 echoes every character + prints `ptz> ` prompt — Pi bridge discards this
- Motion: `vel pan 30.0\n` (no response), `stop all\n`, `estop\n`
- Query: `get position\n` → `pan  : X.XXX °\ntilt : X.XXX °\n`
- bridge.py uses priority queue: estop=0 (never dropped), velocity=1, queries=2

**Camera facts:**
- Sensor: IMX219 (Pi Camera Module 2 / NoIR v2)
- FoV at 1280×720: ~47°H × 28°V (set in config.py — tune empirically if tracking drifts)
- libcamera confirmed working on this Pi. `camera_auto_detect=1` in /boot/firmware/config.txt
- `python3-dlib` is NOT in Bookworm apt — install via pip when face recognition is needed

### Architecture Overview

```
[Raspberry Pi 4]  ← end effector, moves with gimbal
├── libcamera  →  camera capture (NoIR ribbon cable camera)
├── Web server (FastAPI + WebSockets)
│   ├── /stream         — MJPEG (or WebRTC) video feed
│   ├── /ws/control     — WebSocket: browser → Pi → ESP32 motion commands
│   └── /api/...        — REST: settings, detection mode, face enrollment
├── Vision pipeline (OpenCV / MediaPipe / YOLO)
│   ├── Face detection + recognition (local, offline)
│   ├── General object detection (YOLOv8-nano)
│   └── Tracking PID → pan/tilt velocity deltas
└── ESP32 bridge (pyserial over USB-UART, /dev/ttyUSB0)
    └── Wraps existing CLI protocol → no ESP32 firmware changes needed initially

[ESP32]  ← existing firmware, unchanged for Phase 2 start
└── Receives CLI commands from Pi over USB serial
    (e.g. "set vel pan 30.0\n")
```

### Goal-by-Goal Breakdown

**Goal 1 — RPi 4 + IR Camera on end effector**
- Camera in use: IMX219 (Pi Camera Module 2 / NoIR v2) — already confirmed working on this Pi
- FEASIBLE. Main challenges are mechanical: routing the ribbon and power cable
  through the moving gimbal. Options:
  - Slip ring (cleanest, adds complexity/cost)
  - Loose ribbon with strain relief and enough slack for range of motion (simplest)
- Power: 5V/3A to Pi via gimbal wiring. Pi 4 peaks ~3A under load with camera.

**Goal 2 — Real-time video web app**
- FEASIBLE. Best local approach for usable framerates:
  - Start: MJPEG stream via libcamera-apps or OpenCV → simple HTTP server
    - 720p @ 20-30fps is very achievable on Pi 4 on a local network
    - ~100-200ms latency, acceptable for manual PTZ control
  - Upgrade path: mediamtx (formerly rtsp-simple-server) → WebRTC in browser
    - Sub-100ms latency, better for tracking feedback
- Recommended start: MJPEG (simple), migrate to WebRTC only if latency is a problem

**Goal 3 — Gimbal control through the web app**
- FEASIBLE. WebSocket in the browser → FastAPI WS endpoint on Pi →
  pyserial → ESP32 CLI. The existing `set vel pan <n>` and `stop` CLI commands
  are all that's needed. No new ESP32 firmware required to start.
- Control latency (Pi→ESP32 over USB serial): <5ms. Network latency (browser→Pi
  on local WiFi): ~5-20ms. Total loop is well under perceptible lag.

**Goal 4 — Maintain ESP32/Pi segregation**
- RECOMMENDED. ESP32 continues to own real-time motion control and driver UART.
  Pi owns vision, web serving, and user interface. Communication via USB serial.
- The existing CLI protocol is sufficient. If a tighter binary protocol is ever
  needed (e.g., for <1ms command latency or bidirectional telemetry streaming),
  it can be added as a new ESP32 UART environment without breaking existing CLI.
- Red flags to watch: if pyserial drops frames at high control rates, or if
  USB serial adds jitter during tracking. Both are unlikely at ≤50Hz command rate.

**Goal 5 — Simple object detection**
- FEASIBLE on Pi 4.
  - Face detection: OpenCV Haar cascades or MediaPipe Face Detection
    - Haar: ~30fps at 640x480 on Pi 4
    - MediaPipe: ~25fps, more accurate
  - General objects: YOLOv8-nano (Ultralytics, INT8 quantized)
    - ~10-15fps at 640x480 on Pi 4 CPU (no GPU)
    - Sufficient for tracking slow/medium-speed subjects
  - Faster option if needed: Hailo-8L M.2 HAT+ for Pi 5 (not Pi 4 compatible)
    or Coral USB Accelerator (works on Pi 4, adds ~30fps headroom)

**Goal 6 — Object tracking API**
- FEASIBLE. Implementation path:
  1. Detect object/face centroid in frame (pixel x,y)
  2. Compute angular error: (centroid - frame_center) → pan/tilt delta in degrees
     using camera FoV (IMX219 at 1280×720: ~47°H × 28°V — confirmed sensor, set in config.py)
  3. PID controller on the error → velocity command to ESP32
  4. Clamp velocity to motion limits
- Face recognition with local database:
  - `face_recognition` Python library (dlib-based, offline, runs on Pi 4)
  - Enrollment: capture N frames per person, compute face encodings, store in SQLite
  - Recognition: compare live encodings to DB, threshold match → identity label
  - No API calls needed for recognition itself; this is fully local

### Key Risks

1. **Ribbon cable mechanical failure** — The FPC ribbon is fragile over repeated
   flexing. Use a CSI extension cable designed for flex, or a slip ring.
2. **Pi power over moving gimbal wiring** — Voltage drop across long/thin wires;
   use 18AWG minimum, add local 5V regulator on Pi side if needed.
3. **Tracking latency loop** — Detection → PID → serial → driver → motion adds
   ~100-300ms. This limits tracking speed. Start with slow subjects; tune PID tight.
4. **libcamera API changes** — Pi camera stack has changed frequently. Pin the OS
   version (Raspberry Pi OS Bookworm, 2024-11-19 or later) and test before building
   the vision pipeline on top.
5. **USB serial reliability** — Pi USB can reset on power fluctuation. Use a powered
   USB hub or add a TVS diode on VBUS. Implement reconnect logic in the pyserial bridge.
